# API 后端项目架构指南

本文档说明 API 后端新项目可直接采用的通用架构。它关注模块边界、依赖方向、调用链、启动流程和横切能力，不绑定任何具体项目、目录或业务名称。

不包含范围：

- 前端、页面模板、JavaScript、CSS 相关架构
- 认证、授权、Token、权限策略相关设计
- Factory 相关设计
- 某个具体项目、具体目录或具体业务模块的约定

## 一、架构目标

项目架构应优先满足几个目标：

- 业务规则有明确归属，不散落在 Controller 中。
- HTTP 接入、业务处理、领域模型、数据访问、基础设施职责清晰。
- 下层不反向依赖上层，公共能力可复用。
- 可预期业务失败和不可预期系统异常分开处理。
- 新增接口时有稳定路径，不需要重新发明结构。
- 测试、日志、配置、事务、缓存、接口文档等横切能力有固定位置。

## 二、推荐分层

推荐将 API 后端划分为以下层次：

```text
API 层
业务服务层
领域层
数据访问层
基础设施层
外部集成层
```

各层职责如下：

- API 层：接收 HTTP 请求，处理路由、参数默认值、轻量校验和响应包装。
- 业务服务层：承载业务规则、业务校验、业务流程编排、事务边界和业务结果。
- 领域层：定义实体、枚举、值对象、领域默认值和领域级规则。
- 数据访问层：封装数据库访问、实体映射、仓储、查询对象和数据上下文。
- 基础设施层：承载依赖注入、配置、日志、异常处理、缓存、请求管道、接口文档等能力。
- 外部集成层：封装第三方 API、消息队列、对象存储、支付、短信、邮件等外部系统访问。

类库拆分、公共 API、项目引用、Abstractions / Domain / Application / DataAccess 等类库设计规则，见 [CLASS_LIBRARY_GUIDE.md](CLASS_LIBRARY_GUIDE.md)。

## 三、依赖方向

推荐依赖方向：

```text
API 层
    -> 业务服务层
        -> 领域层
        -> 数据访问层
    -> 基础设施层
    -> 外部集成层
```

规则：

- 上层可以依赖下层抽象。
- 下层不应反向依赖上层。
- 领域层不依赖 API 层、HTTP 请求对象、响应包装对象。
- 数据访问层不依赖 Controller。
- 外部集成层不把第三方 SDK 类型泄漏到业务服务层之外。
- 跨 API 应用复用的能力应下沉到业务服务层、领域层或基础设施层。
- 应用特有、不可复用的编排逻辑才保留在 API 应用内部。

## 四、请求调用链

推荐主调用链：

```text
HTTP Request
    -> Middleware
    -> Controller
    -> Service
    -> Repository / DbContext
    -> Database
```

推荐返回链：

```text
Database
    -> Repository / DbContext
    -> ServiceResult / DTO
    -> ApiResult
    -> HTTP Response
```

规则：

- Controller 是薄入口。
- Service 是业务边界。
- Repository / DbContext 是数据访问边界。
- Middleware 处理横切逻辑，不承载业务规则。
- Controller 不直接访问数据库。
- Controller 不直接调用第三方 SDK。

## 五、API 层架构

API 层负责 HTTP 接入，不负责核心业务。

API 层可以做：

- 路由定义。
- 请求参数绑定。
- 分页、排序等默认值补齐。
- 必填参数、格式类轻量校验。
- 调用业务服务。
- 将业务结果转换为 API 结果。

API 层不应做：

- 复杂业务规则。
- 跨表查询拼装。
- 事务控制。
- 直接数据库写入。
- 外部系统调用细节。
- 复杂对象组装。
- 基础设施初始化。

## 六、业务服务层架构

业务服务层是业务规则的主要归属。

业务服务层负责：

- 业务校验。
- 业务流程编排。
- 调用 Repository 或数据上下文。
- 调用外部集成抽象。
- 控制事务边界。
- 返回业务结果对象。
- 处理幂等、状态流转和业务一致性。

业务服务层不应：

- 返回 HTTP 响应对象。
- 依赖 Controller。
- 直接读取 HTTP 请求上下文，除非通过明确抽象传入。
- 把第三方 SDK 类型直接暴露给 API 层。

## 七、领域层架构

领域层应表达稳定的业务概念。

领域层可以包含：

- 实体。
- 值对象。
- 枚举。
- 领域常量。
- 领域默认值。
- 简单领域行为。

领域层应保持纯粹：

- 不依赖 Web 框架。
- 不依赖数据库上下文。
- 不依赖外部 SDK。
- 不依赖 API 返回模型。

如果领域逻辑需要外部数据或复杂流程，应放在业务服务层，而不是让领域对象直接访问基础设施。

## 八、数据访问层架构

数据访问层负责隔离持久化细节。

推荐包含：

- 数据上下文。
- 仓储接口和实现。
- 实体映射。
- 查询扩展。
- 数据迁移或初始化逻辑。

规则：

- 查询场景优先使用无跟踪查询。
- 修改场景使用可跟踪实体或明确更新方法。
- 查询返回应尽量投影为业务需要的数据。
- 不在 Controller 中拼写数据库查询。
- 不把数据库异常直接返回给调用方。
- 数据访问层不决定 HTTP 状态码。

Repository 的职责边界、IQueryable 暴露、查询写入、事务、分页、软删除、审计和并发规则，见 [REPOSITORY_GUIDE.md](REPOSITORY_GUIDE.md)。

## 九、基础设施层架构

基础设施层承载通用技术能力。

常见能力：

- 依赖注入。
- 配置绑定和校验。
- 日志。
- 全局异常处理。
- 请求追踪。
- 缓存。
- 对象映射。
- 接口文档。
- 健康检查。
- 后台任务。

规则：

- 基础设施配置集中在启动阶段或启动扩展中。
- 不把基础设施配置散落在 Controller 或 Service 中。
- 可替换的基础设施能力应通过接口暴露。
- 关键配置应在启动时校验。

## 十、外部集成层架构

外部系统访问应被封装，避免污染业务代码。

推荐做法：

- 为外部系统定义客户端接口。
- 将第三方 SDK 包装在集成层。
- 业务服务依赖集成接口，而不是直接依赖 SDK。
- 外部调用结果转换为项目内部结果对象。
- 对超时、失败、重试、限流和幂等做统一处理。

规则：

- 不让 Controller 直接调用第三方 SDK。
- 不把第三方响应对象直接作为 API 响应返回。
- 外部调用失败要记录日志和 traceId。
- 对可能重复执行的外部调用设计幂等策略。

## 十一、启动架构

推荐启动流程：

```text
应用入口
    -> 读取配置
    -> 注册框架服务
    -> 注册业务依赖
    -> 注册基础设施能力
    -> 校验关键配置
    -> 配置请求管道
    -> 启动应用
```

启动阶段应完成：

- 配置绑定。
- 依赖注入注册。
- 日志配置。
- 数据上下文配置。
- 缓存配置。
- 接口文档配置。
- 全局异常处理接入。
- 请求追踪接入。

启动阶段不应执行复杂业务逻辑。需要初始化的数据，应通过明确的初始化任务处理。

## 十二、结果架构

业务层和 API 层的结果表达应分开：

- 业务服务层使用业务结果对象。
- API 层使用 API 结果对象。
- 可预期业务失败走业务结果对象。
- 不可预期错误走全局异常处理。

结果对象的字段、成功失败格式、分页格式、错误码规则和 HTTP 状态码关系，见 [API_RESULT_GUIDE.md](API_RESULT_GUIDE.md)。

## 十三、横切能力

### 日志

- 关键业务状态变化要记录日志。
- 外部系统调用要记录请求标识和失败原因。
- 异常日志要包含 traceId。
- 不记录敏感信息。

### 配置

- 配置应绑定为强类型对象。
- 密钥不写入源码。
- 关键配置启动时校验。
- 默认值来源要明确。

### 异常

- 可预期业务失败不通过异常表达。
- 不可预期错误由全局异常处理。
- 对外隐藏堆栈和内部异常细节。
- 日志保留完整异常信息。

### 事务

- Service 方法通常是业务事务边界。
- Controller 不控制事务。
- 跨多个写操作时明确事务机制。
- 外部调用和数据库写入混合时考虑补偿。

### 缓存

- 缓存 key 规则要统一。
- 缓存失效策略要明确。
- 不用缓存隐藏业务一致性问题。
- 写操作后要考虑缓存更新或失效。

### 接口文档

- 接口文档应覆盖请求、响应、错误码、分页、幂等规则。
- 文档示例应与统一结果对象一致。
- 已发布接口变更要考虑兼容性。

## 十四、模块边界

划分模块时优先按业务能力，而不是按技术细节。

推荐原则：

- 高内聚：一个模块内部处理同一业务能力。
- 低耦合：模块间通过接口、DTO 或事件协作。
- 依赖稳定：通用能力不依赖具体应用入口。
- 边界清楚：跨模块调用要明确输入输出。

避免：

- 一个通用模块依赖多个具体业务模块。
- 一个 Service 同时处理多个无关业务能力。
- 为了复用而过早抽象。
- 为了方便把所有逻辑放进单个大模块。

## 十五、测试架构

测试也应按层组织：

- API 层：轻量集成测试，验证路由、参数绑定、响应格式。
- 业务服务层：业务规则测试，覆盖成功、失败、边界和状态流转。
- 数据访问层：关键查询和持久化集成测试。
- 外部集成层：使用 mock、stub 或测试替身。
- 基础设施层：验证依赖注入、配置绑定和全局异常处理。

新项目至少应有：

- 容器构建测试。
- 结果对象响应测试。
- 一个典型查询接口测试。
- 一个典型写接口测试。
- 一个业务失败测试。
- 一个全局异常响应测试。

## 十六、新项目落地清单

新项目开始前应确定：

- 分层结构。
- 模块边界。
- 类库拆分和项目引用规则。
- 依赖方向。
- Controller 到 Service 的调用规则。
- 数据访问模式。
- 统一结果对象。
- 错误码规则。
- 全局异常处理。
- 日志和 traceId。
- 配置绑定和校验。
- 事务边界。
- 缓存策略。
- 外部系统封装方式。
- 接口文档规则。
- 测试分层和最小测试集。
